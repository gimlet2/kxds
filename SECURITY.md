# Security Policy

## Supported Versions

We are committed to maintaining security for the following versions of KXDS:

| Version | Supported          |
| ------- | ------------------ |
| 0.1.x   | :white_check_mark: |
| < 0.1.0 | :x:                |

## Reporting a Vulnerability

The KXDS team takes security vulnerabilities seriously. We appreciate your efforts to responsibly disclose your findings.

### How to Report a Security Vulnerability

**Please do NOT report security vulnerabilities through public GitHub issues.**

Instead, please report them via one of the following methods:

1. **GitHub Security Advisory** (Preferred)
   - Navigate to the [Security tab](https://github.com/gimlet2/kxds/security) of this repository
   - Click on "Report a vulnerability"
   - Fill out the form with details about the vulnerability

2. **Email**
   - Send an email to the maintainers (check the repository for contact information)
   - Include the word "SECURITY" in the subject line
   - Provide detailed information about the vulnerability

### What to Include in Your Report

To help us better understand and resolve the issue, please include as much of the following information as possible:

- **Type of vulnerability** (e.g., XXE injection, path traversal, denial of service, etc.)
- **Affected version(s)** of KXDS
- **Step-by-step instructions** to reproduce the vulnerability
- **Proof-of-concept or exploit code** (if possible)
- **Impact** of the vulnerability (what an attacker could achieve)
- **Suggested fix** (if you have one)

### What to Expect

After you submit a vulnerability report:

1. **Acknowledgment**: We will acknowledge receipt of your report within 48 hours
2. **Assessment**: We will assess the vulnerability and determine its severity and impact
3. **Updates**: We will keep you informed about our progress toward a fix
4. **Timeline**: We aim to resolve critical vulnerabilities within 30 days
5. **Disclosure**: Once a fix is available, we will coordinate with you on public disclosure
6. **Credit**: With your permission, we will acknowledge your contribution in our security advisories

### Security Best Practices for Users

When using KXDS in your projects, we recommend:

1. **Keep KXDS up to date**: Always use the latest version to benefit from security fixes
2. **Validate XSD sources**: Only process XSD schemas from trusted sources
3. **Review generated code**: Periodically review the code generated by KXDS
4. **Use secure dependencies**: Keep all dependencies up to date
5. **Follow principle of least privilege**: Run your build process with minimal necessary permissions

### Known Security Considerations

- **XML External Entity (XXE) Processing**: KXDS uses JAXB for parsing XSD schemas. Ensure your XSD files come from trusted sources
- **File System Access**: KXDS reads XSD files from the file system. Validate and sanitize file paths in your configuration
- **Generated Code**: The generated Kotlin code is based on XSD schema definitions. Maliciously crafted schemas could potentially generate problematic code

### Security Updates

Security updates will be released as patch versions (e.g., 0.1.1, 0.1.2) and announced through:

- GitHub Security Advisories
- Release notes
- CHANGELOG.md

## Scope

This security policy applies to:

- The `kxds-hosting` module (KSP processor)
- Example projects (as reference implementations)
- Build scripts and CI/CD configurations

It does not cover:

- Third-party dependencies (report to their respective maintainers)
- User-generated XSD schemas
- Code generated from XSD schemas (responsibility of the schema author)

## Questions?

If you have questions about this security policy, please open a GitHub Discussion or contact the maintainers.

Thank you for helping keep KXDS and its users safe!
